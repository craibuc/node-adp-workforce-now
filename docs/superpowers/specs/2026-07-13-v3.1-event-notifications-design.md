# adp-workforce-now v3.1.0 — Event Notifications + Client.raw

**Date:** 2026-07-13
**Status:** Approved
**Scope:** The ADP event-notification queue behind a dedicated
`EventNotifications` class, plus the public `Client.raw` escape hatch it
requires (response-header access). Semver-minor (3.1.0).

## Context

ADP's event-notification queue delivers subscribed events one message at a
time: `GET /core/v1/event-notification-messages` returns the HEAD of the
queue (204 = empty), with the message's delete handle in the
**`adp-msg-msgid` response header** — and the next message only becomes
available after the current one is DELETEd
(`DELETE /core/v1/event-notification-messages/{id}`). Deleting after
processing gives at-least-once semantics naturally. The production flow
being replaced loops up to 50 iterations processing whatever the queue
holds; its recorded Python logic (returns body with the header id injected,
`None` on 204, `raise_for_status` otherwise) is the behavioral reference.

`Client.request()` returns only the parsed body, so header access needs new
plumbing — exposed publicly per the library's escape-hatch philosophy.

## `Client.raw`

```typescript
raw(method: string, path: string, data?: unknown):
  Promise<{ status: number; headers: Headers; body: unknown }>
```

- Identical semantics to `request()`: lazy auth, mTLS, exactly one 401
  force-refresh retry, `raiseForAdp` thrown on non-2xx.
- 204: resolves with `body: undefined` but real `status`/`headers` (unlike
  `request()`, which collapses 204 to `undefined`).
- `request()` is refactored into a thin wrapper over `raw()` — one
  implementation of the auth/retry/error pipeline. No behavior change for
  existing callers (pinned by the existing suite).

## `EventNotifications` (new `src/event-notifications.ts`)

Reached as `client.eventNotifications` (constructed in the `Client`
constructor like `worker`).

```typescript
interface EventNotificationMessage {
  /** Delete handle from the adp-msg-msgid response header. */
  messageId: string;
  /** ADP's response body, untouched (typically an event envelope). */
  payload: unknown;
}

class EventNotifications {
  next(): Promise<EventNotificationMessage | null>;
  delete(messageId: string): Promise<unknown>;
}
```

- `next()`: `raw('GET', '/core/v1/event-notification-messages')`.
  - 204 → **`null`** (deliberate divergence from the library's usual
    `undefined`: this class's primary consumers are flow steps whose return
    values are JSON-serialized, and `null` survives that round trip while
    `undefined` vanishes — the queue-empty signal must be representable).
  - 200 → `{ messageId, payload }` where `messageId` comes from the
    `adp-msg-msgid` header and `payload` is the parsed body, untouched
    (transport metadata is NOT injected into the payload, unlike the legacy
    Python which wrote it into `events[0]`).
  - 200 with the header missing → throw
    `Error('event-notification response is missing the adp-msg-msgid header')`
    — a message that cannot be acknowledged would silently jam the queue.
  - Repeated `next()` without `delete()` returns the SAME message — the
    queue only advances on delete. Documented prominently.
- `delete(messageId)`:
  `raw('DELETE', '/core/v1/event-notification-messages/{id}')`
  (percent-encoded), returning the response body pass-through. Recorded
  production behavior (legacy Python): 200 echoes the deleted
  event-notification record(s) as JSON; other 2xx statuses have no body —
  return **`null`** then (same JSON-representable convention as `next()`,
  matching the Python's `None`). Errors surface via `raiseForAdp` as usual.

## Consumption pattern (documented in README)

```typescript
// Process up to 50 queued notifications (the production-flow pattern):
for (let i = 0; i < 50; i++) {
  const message = await client.eventNotifications.next();
  if (message === null) break;              // queue empty
  await handle(message.payload);            // your logic
  await client.eventNotifications.delete(message.messageId); // ack -> advances queue
}
```

Windmill note: each iteration can also be its own flow-loop step — the queue
itself holds the position, so unlike `worker.search().page(n)` there is no
index to carry between iterations.

## Testing

- `tests/helpers/fake-transport.ts`: canned responses gain an optional
  `headers?: Record<string, string>` merged into the `Response` (additive —
  existing tests unaffected).
- Unit (fake transport): `raw()` returns status/headers/body and still
  throws typed errors on non-2xx; `raw()` 204 keeps headers; the existing
  suite pins that `request()`-through-`raw()` refactor changed nothing
  (401-retry, 204, masked, error mapping tests already exist).
  `next()`: maps header + body; 204 → `null`; missing header → the specific
  Error; repeated-message behavior is ADP-side (not unit-testable) but the
  same-URL statelessness is asserted. `delete()`: method/URL wire format,
  percent-encoding, body pass-through (echoed-event 200 and bodyless-204 -> null variants).
- Live (env-gated, added to the integration suite): one `next()` call —
  either `null` (empty queue) or a shape assertion (`messageId` non-empty
  string, `payload` truthy) **without deleting** (the queue is left exactly
  as found; deleting the head of the production queue from a test is not
  acceptable). `delete()`'s echo behavior is verified when the migrated
  flow first runs, not by the gate.
