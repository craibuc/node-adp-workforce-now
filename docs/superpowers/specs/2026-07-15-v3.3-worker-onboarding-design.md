# adp-workforce-now v3.3.0 — Worker Onboarding (Applicant Onboarding v2)

**Date:** 2026-07-15
**Status:** Approved
**Scope:** `Worker.onboard()` wrapping `POST /hcm/v2/applicant.onboard`, plus
the pipeline generalizations it requires (per-event path registry, per-event
blocking policy, wrapperless-envelope flattening, a fourth error shape).
Semver-minor (3.3.0).

## Recorded reference (production Python, verbatim behavior)

- `POST https://api.adp.com/hcm/v2/applicant.onboard` with body root
  `{ "applicantOnboarding": { … } }` — NO `events[]` wrapper.
- **This family's wire dialect differs from the events family** (all
  live-recorded; do not "normalize" them):
  - code objects are `{"code": …}` (not `codeValue`) — e.g.
    `onboardingTemplateCode: {code}`, `genderCode: {code}`,
    `hireReasonCode: {code}`;
  - EXCEPT `communication` entry nameCodes, which use `{"codeValue": …}`
    ("Home Phone", "Personal Cell", "Personal E-mail");
  - `governmentIDs: [{ "id": ssn, "nameCode": {"code": "SSN"} }]` — `id`,
    not `idValue`;
  - address uses `subdivisionCode` (not `countrySubdivisionLevel1`);
  - `payrollGroupCode` is a PLAIN STRING (matches the meta path);
  - `onboardingStatus: { statusCode: { code: "inprogress" } }` is sent
    verbatim;
  - phones are split into `countryDialing "1"` + 3-digit `areaDialing` +
    remaining `dialNumber` after stripping non-digits.
- Headers in the recorded script: `Accept` and `Content-Type` both
  `application/json;masked=false`. The library uses its standard headers
  (Accept per the client's `masked` option; `Content-Type:
  application/json`); if the live proof ever 400s on content type, mirroring
  the recorded header is the first fix to try.
- Error responses (400/500) use a FOURTH shape:
  `_confirmMessage.messages[0].messageText`. 403 uses the known
  `response.applicationCode` shape.
- Success: 200/201/204; body JSON or empty.
- Latent bug noted in the legacy Python (patches
  `body["applicantTaxProfile"]…` although the root is `applicantOnboarding`
  — would KeyError if its condition ever fired); the TS version implements
  the additional-tax amounts correctly as optionals.
- Tenant-specific payload quirks (e.g. a `DataControl` custom code field
  whose value carries trailing spaces) are NOT baked into the library —
  they flow through `payroll.customCodeFields` / `overrides` from the
  caller's own script.

## Live meta evidence (tenant, 2026-07-15)

`GET /hcm/v2/applicant.onboard/meta` parses to exactly 5 rules, all
`optional:false`, none with code lists:
`applicantOnboarding.onboardingTemplateCode`,
`…applicantPersonalProfile.birthName.givenName`, `…birthName.familyName`,
`…applicantWorkerProfile.hireDate`,
`…applicantPayrollProfile.payrollGroupCode`. (Raw rule paths carry a
leading `/meta/` segment — see parsing change below.)

## Pipeline generalizations

### Per-event path + blocking registry (in `src/meta.ts`)

```typescript
interface EventRoute {
  postPath: string;              // e.g. '/hcm/v2/applicant.onboard'
  metaPath: string;              // e.g. '/hcm/v2/applicant.onboard/meta'
  /** Issue codes that BLOCK a post. Default: ['codeList']. */
  blocking: ReadonlyArray<ValidationIssue['code']>;
}

function eventRoute(event: string): EventRoute;
// 'applicant.onboard' -> { '/hcm/v2/applicant.onboard', …/meta,
//                          blocking: ['codeList', 'required'] }
// everything else     -> { `/events/hr/v1/${event}`, …/meta,
//                          blocking: ['codeList'] }
```

`Worker.eventMeta` and `Worker.postEvent` consult the registry for paths;
`postEvent`'s blocking filter uses `route.blocking` (both the pre-POST check
and the 400 self-heal re-check). `required` blocking for
`applicant.onboard` is evidence-justified: the meta is small and clean, and
all five required paths appear in the recorded working payload — the
overdeclare risk that forced codeList-only elsewhere does not apply. It
directly implements the requirement that the body be validated against the
meta before the POST is attempted.

### Meta parsing (`parseEventMeta`)

Leading `meta` path segments are dropped as structural (like array
indices), so the onboard rules become
`transform:/applicantOnboarding/…`. No existing rule path contains a
`meta` segment (the events-family scoping cuts at `transform(s)`), so this
is backward-invisible — pinned by the existing meta tests staying green.

### Wrapperless envelopes (`flattenEnvelope`)

When an envelope has NO `events` array at its root, the whole object is
flattened from the root under the `transform:` scope. Existing `events[]`
envelopes behave exactly as before. With both changes, the onboard
envelope's paths line up with the meta's rules (container-required rules
satisfied by leaves beneath them, per the existing v3 semantics; a missing
`hireDate` produces a blocking `required` issue naming the path).

## Fourth error shape (`src/errors.ts`)

`extract()` gains, after the existing `exceptionMessages` shape:
`_confirmMessage.messages[0].messageText` → `adpMessage` (plus
`messages[0].messageCode`-style code if present as a string under a `code`
or `messageCode` key; absent → undefined). Synthetic-fixture test; ordering
before the OAuth shape.

## `Worker.onboard(params)`

```typescript
interface OnboardParams {
  /** Tenant onboarding template. */
  onboardingTemplateCode: string;
  personal: {
    givenName: string;
    familyName: string;
    middleName?: string;
    birthDate?: string;            // YYYY-MM-DD
    genderCode?: string;           // also mirrored into genderReportingDetails
    raceCode?: string;
    raceIdentificationMethodCode?: string;
    ethnicityCode?: string;
    languageCode?: string;
    ssn?: string;
    address?: {
      lineOne: string; lineTwo?: string; cityName: string;
      stateCode: string; postalCode: string; countryCode?: string; // default 'US'
    };
    homePhone?: string;            // free-form; digits extracted, split 3/rest
    mobilePhone?: string;
    email?: string;
  };
  worker: {
    hireDate: string;              // YYYY-MM-DD
    hireReasonCode?: string;
    jobCode?: string;
    workerTypeCode?: string;
    businessUnitCode?: string;     // homeOrganizationalUnits BusinessUnit
    homeDepartmentCode?: string;   // homeOrganizationalUnits HomeDepartment
    reportsToPositionID?: string;
    eeoClassificationCode?: string;
    eeocClassificationCode?: string;
    managementPositionIndicator?: boolean;  // default false
  };
  payroll: {
    payrollGroupCode: string;      // plain string on the wire
    payCycleCode?: string;
    payrollScheduleGroupCode?: string;
    /** Tenant custom code fields (e.g. a DataControl entry) — passed through verbatim. */
    customCodeFields?: Array<{ nameCode: string; code: string }>;
  };
  tax?: {
    federal?: {
      taxFilingStatusCode?: string;
      deductions?: number;         // allowanceTypeCode Deductions
      dependents?: number;         // allowanceTypeCode Dependents
      additionalTaxAmount?: number;
      multipleJobIndicator?: boolean;  // default false
    };
    state?: {
      workedInStateCode?: string;  // stateIncomeTaxInstructions workedInJurisdiction
      livedInStateCode?: string;   // second instruction, livedInJurisdiction
      taxFilingStatusCode?: string;
      taxAllowanceQuantity?: number;
      additionalTaxAmount?: number;
    };
  };
  /** Deep-merged over the generated applicantOnboarding object last — the
   *  tenant escape hatch for anything not modeled above. */
  overrides?: Record<string, unknown>;
}

onboard(params: OnboardParams): Promise<unknown>;  // response body, or null when empty
```

- Builds the envelope per the recorded dialect (all quirks above);
  `onboardingStatus` "inprogress" always sent; sections and entries are
  OMITTED when their params are absent (divergence from the recorded
  script, which sent null-filled phone entries — omission is cleaner; the
  first live flow run is the gate for this choice).
- Routes through `postEvent('applicant.onboard', envelope)` — auth, meta
  validation (required+codeList blocking per the registry), 400 self-heal,
  typed errors all inherited. The 5 meta-required fields are ALSO
  compile-time required in `OnboardParams` — runtime validation catches
  type-defeating callers and future template-driven meta changes.
- `overrides` deep-merge (small internal util; objects merge recursively,
  arrays and scalars replace) applies after generation, before validation —
  so validation sees the final body.
- Returns the response body pass-through, `null` when ADP returns no body
  (flow-step JSON convention).

## Testing

- Registry: route resolution for `applicant.onboard` vs default events;
  blocking policy differences pinned (a `required` issue blocks onboard but
  NOT a default event — synthetic metas).
- Meta parsing: leading-`meta` segment drop (synthetic onboard-shaped
  fixture); existing meta tests unchanged (backward-invisible pin).
- flattenEnvelope: wrapperless root flatten; existing envelope tests
  unchanged.
- Errors: `_confirmMessage` shape extraction.
- `onboard()` wire tests (fake transport, validation on with synthetic
  onboard meta): full envelope against the recorded dialect (code vs
  codeValue vs id spellings; subdivisionCode; plain-string
  payrollGroupCode; phone digit-split; status inprogress; EEO/EEOC
  classifications; org units); omission behavior (no phones → no
  communication entries; no tax → no applicantTaxProfile); missing
  `hireDate` via type-defeat → blocking `required` EventValidationError
  with NO POST issued; overrides deep-merge wins and is validated;
  customCodeFields pass-through.
- Live gate: onboard meta fetch (with the existing retry) asserting the 5
  required rules parse to the expected paths; envelope path-coverage
  against the live meta (all 5 required paths present in a representative
  generated envelope). NO live onboard call — creating a visible
  in-progress onboarding is not acceptable from a test; **the first
  controlled run of the migrated onboarding flow is the wire-format
  proof** (same pattern as the queue delete-echo).
