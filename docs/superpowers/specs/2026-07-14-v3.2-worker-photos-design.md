# adp-workforce-now v3.2.0 — Worker Photos (Binary Transport)

**Date:** 2026-07-14
**Status:** Approved
**Scope:** `Worker.getPhoto` / `Worker.setPhoto` and the binary
request/response plumbing they require. Semver-minor (3.2.0).

## Recorded references (production Python, verbatim behavior)

- **Read:** `GET /hr/v2/workers/{aoid}/worker-images/photo` — plain
  authenticated GET (the legacy script sends `Accept: application/json`,
  presumed inert; the endpoint serves the image itself — the live gate
  records the actual response content-type).
- **Upload:** `POST /events/hr/v1/worker.photo.upload` — multipart/form-data
  with exactly two parts (names are load-bearing):
  - part `json`: the envelope
    `{"events":[{"data":{"eventContext":{"worker":{"associateOID": …}}}}]}`
    (no transform, no event codes);
  - part `datafile`: the image bytes (legacy Python passes an opened file;
    `requests` supplies the filename from the file handle).
- Upload error shapes observed: 400 `confirmMessage…messageTxt`, 403
  `response.applicationCode.message` — both already handled by
  `raiseForAdp`; the legacy scripts' hand-parsing becomes unnecessary.
- `GET /events/hr/v1/worker.photo.upload/meta` exists (API explorer);
  fetched live by the gate as corroborating evidence.

## Transport widening

- `TransportInit.body: string | Uint8Array`.
  - Node adapter: `Content-Length` via `Buffer.byteLength(body)` (correct
    for both types); `req.write(Buffer.from(body))`.
  - Bun adapter: `fetch` accepts `Uint8Array` bodies natively — no change
    beyond the type.
- `Client` gains INTERNAL plumbing (not exported, not public API) that the
  photo methods use — a generalized request core that:
  - accepts an optional caller-controlled `Content-Type` (the multipart
    boundary header; today `send()` hardcodes `application/json` whenever a
    body is present);
  - can return the response as bytes (`Uint8Array` via
    `response.arrayBuffer()`) instead of text-parsing.
  - Same lazy auth, exactly-one 401 retry, and `raiseForAdp` semantics.
  - `raw()` remains the public JSON escape hatch; a public binary variant
    is deliberately deferred (YAGNI) and can ship semver-minor when
    something needs it.

## Multipart builder (zero-dep)

A small internal helper (in the photos module) that assembles
`multipart/form-data` bytes with a generated boundary:

- part `json` — `Content-Disposition: form-data; name="json"`, body = the
  envelope JSON string;
- part `datafile` — `Content-Disposition: form-data; name="datafile";
  filename="<filename>"` + `Content-Type: <contentType>`, body = image
  bytes.

The request `Content-Type` is `multipart/form-data; boundary=<boundary>`.
The exact per-part headers `requests` emits are not fully knowable from the
recorded sample; the opt-in live upload gate is the authority — if ADP
rejects this construction, the gate fails before any consumer sees it.

## Methods (on `Worker`)

```typescript
interface WorkerPhoto {
  contentType: string;   // from the response Content-Type header
  bytes: Uint8Array;
}

interface SetPhotoParams {
  associateOID: string;
  /** Image bytes, or a base64 string (auto-detected — a string is decoded). */
  image: Uint8Array | string;
  /** datafile part Content-Type. Default "image/jpeg". */
  contentType?: string;
  /** datafile part filename. Default "photo.jpg". */
  filename?: string;
}

getPhoto(aoid: string): Promise<WorkerPhoto | null>;
setPhoto(params: SetPhotoParams): Promise<unknown>;
```

- `getPhoto`: binary GET (aoid percent-encoded). **`null` on 404 and 204**
  — a missing photo is a normal state, not an error (consistent with the
  event-notifications `null` convention); every other error throws as
  usual (`NotFoundError` is caught and mapped to `null` for this method
  only).
- `setPhoto`: builds the recorded envelope + multipart body and POSTs.
  **Bypasses `postEvent`** (multipart, not the JSON pipeline) but keeps the
  full auth/retry/typed-error stack. Returns ADP's response JSON
  pass-through.
- Base64 detection in `image`: `typeof image === 'string'` → decoded via
  `Buffer.from(image, 'base64')` (same convention as PEM handling; callers
  passing bytes are untouched).

### Meta-driven size preflight + content-type sniff

Live evidence (tenant meta, 2026-07-14): the upload meta declares
`transform:/worker/photo/imageSize` `{optional:false, pattern:"^[0-9]*$",
maxLength:200000}` — ADP encoding the max photo byte size (~200 KB) in the
`maxLength` field. (Also corroborates the overdeclare pattern: the meta
marks `transform/worker/photo` required, yet the recorded working request
sends no transform at all.) There are NO dimension rules in the meta, so
in-library pixel scaling would validate a rule that does not exist — actual
resizing is a README recipe (`sharp` in the consumer's script), not library
scope.

- Before uploading, `setPhoto` fetches `eventMeta('worker.photo.upload')`
  (existing cache + negative cache; FAIL-OPEN if unavailable, per the
  established convention) and reads the `transform:/worker/photo/imageSize`
  rule's `maxLength`. If the image byte length exceeds it, throw a clear
  `Error` naming both numbers (e.g. "photo is 312450 bytes; tenant limit is
  200000 — resize before uploading") WITHOUT making the upload request.
- `contentType` default is derived from magic bytes when not supplied:
  `FF D8 FF` → `image/jpeg`; `89 50 4E 47` → `image/png`; anything else →
  `image/jpeg` (the historical default). An explicit `contentType` always
  wins.

## Testing

- Transport: unit tests for `Uint8Array` bodies through both adapter
  paths (fake fetch capture for Bun; the real local mTLS server for Node —
  binary echo asserting byte-accurate `Content-Length` and round-trip).
- Multipart builder: byte-level assertions — boundary syntax, both part
  headers (names `json`/`datafile`, filename, part content-type), CRLF
  framing, terminal boundary.
- `getPhoto` (fake transport): maps content-type + bytes; 404 → null;
  204 → null; non-photo errors still throw.
- `setPhoto` (fake transport): request Content-Type carries the boundary;
  decoded-base64 and raw-bytes inputs produce identical bodies; envelope
  JSON exact; defaults applied; overrides honored.
- Preflight (fake transport, synthetic meta with a small imageSize limit):
  oversized image → clear Error naming both numbers, NO upload request
  issued (call-count pinned); within-limit → upload proceeds; meta
  unavailable → fail-open upload proceeds. Sniff: JPEG magic → image/jpeg
  part header; PNG magic → image/png; explicit contentType overrides both.
- **Live gate:** `getPhoto(ADP_ASSOCIATE_OID)` runs in the standard
  env-gated suite — read-only; both outcomes pass (null, or bytes with the
  actual content-type logged). **Upload is a visible write** (replaces the
  worker's photo in ADP), so the live upload test is opt-in behind
  `ADP_PHOTO_TEST_AOID`: a tiny generated image uploaded to that test
  worker, then `getPhoto` re-read to confirm round-trip. That opt-in run is
  the release checklist item for the multipart format. The standard gate
  also fetches `eventMeta('worker.photo.upload')` (read-only) and asserts
  it parses.
