# adp-workforce-now v2.1 — Event-Meta Layer + New Events Design

**Date:** 2026-07-13
**Status:** Approved
**Scope:** Generic event metadata (fetch, cache, client-side payload
validation with 400 self-heal) and four new event methods. First of three
planned expansions: v2.1 (this spec) → v2.2 (notification queue + worker
search) → v2.3 (worker photos).

## Context

v2.0.0 ships seven wrapped endpoints and a `Client.get`/`post` escape hatch.
ADP event POSTs are validated server-side against tenant-configurable
validation tables; the failure mode is an opaque 400 after the round trip.
Every event exposes `GET /events/hr/v1/{event}/meta` describing its
constraints. Metas are tenant-level (not per-worker) and change rarely —
safe to cache for hours. This release builds that meta layer once and adds
the next four events on top of it.

Decisions made during design review:

- **Pipeline over utility:** validation lives in a private
  `Worker.postEvent()` pipeline all event methods route through — not an
  opt-in helper consumers must remember to call.
- **`validateEvents` defaults ON**, including for the existing
  `hire`/`rehire`/`terminate` (behavioral change from 2.0: a bad reason code
  now fails client-side with a readable error; `validateEvents: false`
  restores 2.0 behavior).
- **`eventMeta(event)` replaces `hireMeta()`**; `hireMeta` remains as a
  `@deprecated` delegate until 3.0 (semver).
- **Custom fields: string type only** for now; the meta layer makes sibling
  types (code/date/number/indicator) mechanical to add later.
- Version: **2.1.0**, released via the tag-push provenance workflow (its
  first live run).

## New module: `src/meta.ts`

Everything meta-related lives here; `worker.ts` consumes it.

```typescript
/** String-literal union of events the library wraps. */
type SupportedEvent =
  | 'worker.hire'
  | 'worker.rehire'
  | 'worker.work-assignment.terminate'
  | 'worker.work-assignment.base-remuneration.change'
  | 'worker.legal-name.change'
  | 'worker.person.custom-field.string.change'
  | 'worker.leave.absence.request';

/** Parsed, flattened view of ADP's meta response. */
interface EventMeta {
  event: SupportedEvent;
  /** JSON-pointer-ish path -> constraints, as ADP's meta reports them. */
  rules: Map<string, FieldRule>;
  raw: unknown;               // untouched meta body
  fetchedAt: number;          // epoch ms, for cache TTL
}

interface FieldRule {
  optional?: boolean;
  readOnly?: boolean;
  hidden?: boolean;
  codeValues?: string[];      // from codeList.listItems[].codeValue
  pattern?: string;
  minLength?: number;
  maxLength?: number;
}

interface ValidationIssue {
  path: string;               // envelope path the issue applies to
  code: 'required' | 'readOnly' | 'hidden' | 'codeList' | 'pattern' | 'length';
  message: string;            // human-readable, includes allowed values for codeList
}

function parseEventMeta(event: SupportedEvent, raw: unknown): EventMeta;
function validateEnvelope(envelope: unknown, meta: EventMeta): ValidationIssue[];
```

`validateEnvelope` checks, per meta rule: required fields present
(`optional: false`), `readOnly`/`hidden` fields absent, values within
`codeValues`, `pattern` match, and length bounds. It returns issues — it
never throws. Fields present in the envelope but absent from the meta are
NOT flagged (metas are not exhaustive; false positives would block valid
posts). Unparseable metas degrade gracefully: `rules` is empty and
validation passes everything (fail-open — the server remains the authority).

## Validation error

```typescript
class EventValidationError extends Error {
  readonly event: string;
  readonly issues: ValidationIssue[];
}
```

Deliberately NOT an `AdpError` subclass: nothing was sent, there is no HTTP
status. Exported from the main entry. Message format:
`worker.rehire failed client-side validation (2 issues): <first issue>…`.

## Meta cache and `eventMeta()`

- `Worker.eventMeta(event: SupportedEvent, options?: { forceRefresh?: boolean }): Promise<EventMeta>`
  — `GET /events/hr/v1/{event}/meta`, parsed via `parseEventMeta`.
- In-memory cache on the `Client` instance, keyed by event, **12 h TTL**.
  Metas are tenant-level, so per-client caching is correct; no pluggable
  store (unlike tokens, metas are not a cross-language interop contract).
- `Worker.hireMeta()` becomes `@deprecated` and delegates to
  `eventMeta('worker.hire')`, returning the RAW meta body (its 2.0 contract)
  — not the parsed form.

## The pipeline

**Public** `Worker.postEvent(event: SupportedEvent | (string & {}), envelope: unknown)`
— public so unwrapped `worker.*` events get the validated pipeline as an
escape hatch (autocompletes the supported events; accepts any event string,
for which meta is fetched from the same uniform endpoint pattern). Design
review decision: worker event actions live on `Worker` (they are actions on
a worker); no generic `Events` namespace. The v2.2 notification queue is a
SEPARATE class (see Out of scope). Pipeline behavior:

1. If `client` option `validateEvents` (default `true`):
   `eventMeta(event)` (cached) → `validateEnvelope` → non-empty issues throw
   `EventValidationError`.
2. `POST /events/hr/v1/{event}` via the existing `Client.post` (auth, mTLS,
   401 retry, `raiseForAdp` all apply unchanged).
3. **Self-heal on 400** (only when validation is on): catch
   `BadRequestError` → `eventMeta(event, { forceRefresh: true })` →
   re-validate the same envelope. If it NOW fails, throw
   `EventValidationError` (the cache was stale — tenant edited a validation
   table); otherwise rethrow the original `BadRequestError` untouched. The
   POST is never retried — self-heal improves the error, it does not
   resubmit.

Existing `hire`, `rehire`, `terminate` are re-routed through `postEvent`
with their envelopes unchanged.

## New event methods

Naming follows the README's `changeBaseRemuneration` convention. All return
the raw ADP response (`Promise<unknown>`), like the existing events.

```typescript
interface ChangeBaseRemunerationParams {
  associateOID: string;
  workAssignmentID: string;       // workAssignments[].itemID
  /** YYYY-MM-DD; chosen by caller (usually next pay period start). */
  effectiveDate: string;
  rateType: 'H' | 'D' | 'S';      // hourly | daily | salary (pay-period)
  amount: number;
  currencyCode?: string;          // default 'USD'
  /** Tenant Compensation Change Reasons code — validated against meta. */
  eventReasonCode: string;
}

interface ChangeLegalNameParams {
  associateOID: string;
  givenName: string;
  familyName: string;
  middleName?: string;
  effectiveDate: string;          // YYYY-MM-DD
  eventReasonCode?: string;
}

interface ChangeCustomFieldStringParams {
  associateOID: string;
  /** The custom field instance itemID on the worker record. */
  itemID: string;
  stringValue: string;
  effectiveDate?: string;         // YYYY-MM-DD
}

interface RequestLeaveAbsenceParams {
  associateOID: string;
  startDate: string;              // YYYY-MM-DD
  expectedReturnDate?: string;    // YYYY-MM-DD
  reasonCode: string;
  comments?: string;
}
```

### Envelope sources and verification discipline

- `changeBaseRemuneration`: wire format fully documented from prior
  research — `{ events: [{ data: { eventContext: { worker: { associateOID,
  workAssignment: { itemID } } }, transform: { effectiveDateTime,
  eventReasonCode: { codeValue }, worker: { workAssignment:
  { baseRemuneration: { hourlyRateAmount | dailyRateAmount |
  payPeriodRateAmount: { nameCode: { codeValue: 'H'|'D'|'S' }, amountValue,
  currencyCode } } } } } }] }`. This API family uses **`amountValue`**
  (matching Workers v2 GET reads), not the legacy PUT's `amount`.
  Backdating does not auto-calculate retro pay.
- `changeLegalName`, `changeCustomFieldString`, `requestLeaveAbsence`:
  envelopes are drafted from the API Explorer's documented shapes and MUST
  be verified against the live event metas before first production use. The
  implementation plan includes an env-gated live test that fetches each
  event's meta and asserts our envelope paths appear in it; the release
  checklist requires that test green against a real tenant.

## Testing

- Meta parsing/validation: synthesized meta fixtures under
  `tests/fixtures/meta/` exercising each rule type (required, readOnly,
  codeList, pattern, length), plus a sanitized recorded meta if one can be
  captured from a real tenant.
- Pipeline (fake transport): validation pass; validation failure (no POST
  issued — assert call count); 400 self-heal where the refreshed meta
  newly rejects the envelope (`EventValidationError`) and where it still
  accepts it (original `BadRequestError` rethrown, no re-POST);
  `validateEvents: false` bypass; meta cache hit/expiry (injectable clock
  not required — use a short TTL override for tests).
- Envelope wire-format tests per new method (parse the recorded POST body),
  same style as the existing event tests.
- `hireMeta()` deprecation: still returns the raw meta body.
- Env-gated live test: fetch all supported event metas from the real tenant;
  assert each parses and that each drafted envelope's field paths exist in
  its meta.

## Out of scope (recorded decisions for later specs)

- **v2.2 — queue + search:** a separate `EventNotifications` class reached
  as `client.eventNotifications`, with `next()` (GET
  `/core/v1/event-notification-messages` — one message per call, 204 =
  empty queue → `undefined`, delete handle in the `adp-msg-msgid` response
  header — requires exposing response headers through the client) and
  `delete(messageId)` (DELETE `/core/v1/event-notification-messages/{id}`); `findByName`
  (server-side single-field `$filter` — ADP rejects compound `and` filters
  on some tenants; when both names are given, filter on familyName
  server-side and match givenName client-side) and `findBySSN` (client-side
  page scan with `masked: false`; requires unmasked scope; O(worker count),
  documented as such).
- **v2.3 — photos:** `GET /hr/v2/workers/{aoid}/worker-images/photo`
  (binary response) and `POST /events/hr/v1/worker.photo.upload` (binary/
  multipart body) — both require widening the transport body/response
  contract beyond strings.
- Custom-field sibling types (code/date/number/indicator/percentage).
- Roadmap-only (no version assigned): `worker.work-assignment.modify`,
  personal/business email + mobile change events, legal/personal address
  change events, `worker.pay-distribution.change` (direct deposit — ships
  with extra validation care when scheduled).
